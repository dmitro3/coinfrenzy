End-to-End Flow

This is the complete lifecycle from affiliate signup to payout. Read this first.

1
Affiliate creates a CoinFrenzy.com account. They choose a unique username (e.g., coolcreator123) visible on their profile page.
2
Affiliate registers on frenzycreator.com. During signup they enter their CoinFrenzy username. frenzycreator.com calls GET /affiliate/user/{username} to verify the username exists on CoinFrenzy.com. If non-200, signup is blocked.
3
Admin approves the affiliate. On approval, frenzycreator.com calls POST /affiliate to register this affiliate on CoinFrenzy.com's side. If non-200, approval does not proceed.
4
Affiliate creates campaigns on frenzycreator.com. Each campaign generates a unique promo code and referral link. Example: promo code BRYCE → link coinfrenzy.com?code=bryce. frenzycreator.com calls POST /affiliate/promocode to sync the code to CoinFrenzy.com. If non-200, the campaign is not created.
5
Affiliate shares their link or promo code. They can share the referral link (for bio, ads, DMs) or just tell their audience to enter the promo code at signup (e.g. "Use code BRYCE for special rewards").
6
New player signs up on CoinFrenzy.com. Either via the ?code= link OR by entering the promo code during signup. CoinFrenzy.com sends a webhook to frenzycreator.com with the player data, linking them to the affiliate who owns that code.
7
Player plays on CoinFrenzy.com. GGR accrues per player. CoinFrenzy.com calculates NGR as GGR − (|GGR| × 0.25). The 25% deduction always works against the affiliate — reducing profit when positive, increasing loss when negative.
8
Weekly: CoinFrenzy.com pushes NGR data for all referred players to our Supabase. One record per player per week.
9
Admin reviews and triggers payout. The admin panel aggregates NGR per affiliate, calculates commission, and clicks "Pay". frenzycreator.com calls POST /affiliate/payout to credit the affiliate's Lightning Bolt rewards on CoinFrenzy.com (claimable from the lightning icon at the top of the site). If non-200, payout is not recorded.

Campaigns & Promo Codes

Critical: How referral tracking works

Affiliates do NOT get an auto-generated referral link at signup. Instead, they create campaigns on frenzycreator.com, each with a unique promo code. The promo code is the referral identifier — it powers both the link and the manual code entry at signup. CoinFrenzy.com must handle both methods.

How Campaigns Work

  1. Affiliate logs into frenzycreator.com and navigates to the Campaigns page
  2. They create a campaign (e.g., name: "TikTok Bio", promo code: BRYCE)
  3. The system generates a referral link: coinfrenzy.com?code=bryce
  4. The system also displays the promo code: BRYCE
  5. Promo codes are globally unique — no two affiliates can use the same code
  6. Affiliates can create unlimited campaigns (one per platform, ad, page, etc.)

Referral Link Format

// The referral link format (generated by frenzycreator.com)
https://coinfrenzy.com?code={promo_code}

// Examples:
https://coinfrenzy.com?code=bryce
https://coinfrenzy.com?code=tiktok-vip
https://coinfrenzy.com?code=ig-story

Two Ways a Player Can Be Referred

Method 1: Referral Link

Player clicks coinfrenzy.com?code=bryce. CoinFrenzy.com reads the ?code= route, extracts the promo code, stores it in a cookie/session, and attributes the signup to the affiliate who owns that code.

Method 2: Promo Code at Signup

Player goes directly to coinfrenzy.com and enters promo code BRYCE in a field during signup. CoinFrenzy.com resolves the code to the affiliate. This is for affiliates who prefer saying "use code BRYCE" instead of sharing a link.

How Promo Codes Are Synced to CoinFrenzy.com

When an affiliate creates, updates, or deletes a campaign on frenzycreator.com, we call the CoinFrenzy API to keep the main site in sync:

  1. Create: POST https://dev2-cron.coinfrenzy.com/api/v1/frenzy-creator/promocode — called when a new campaign is created. If non-200, the campaign is not saved.
  2. Update: PUT https://dev2-cron.coinfrenzy.com/api/v1/frenzy-creator/promocode/{id} — called when a campaign is edited.
  3. Delete: DELETE https://dev2-cron.coinfrenzy.com/api/v1/frenzy-creator/promocode/{id} — called when a campaign is removed.

This means CoinFrenzy.com maintains its own copy of all promo codes and can resolve them locally — no need to query our Supabase.

What CoinFrenzy.com Must Do

  1. Handle the ?code={code} URL query parameter. When a user visits coinfrenzy.com?code=bryce, store the promo code bryce in a cookie or session so it persists through signup.
  2. Add a "Promo Code" input field on the signup page. Players can optionally enter a code. If the ?code= cookie is present, auto-fill this field.
  3. Resolve the promo code to an affiliate. Look up the code in the local promo code records (synced from frenzycreator.com via the API).
  4. Send a webhook to frenzycreator.com when a player completes registration, including promo_code_used and source (see Player Registration).

Promo Code Rules

Promo codes are case-insensitive (stored lowercase, displayed uppercase). They contain only letters, numbers, and dashes. Max 30 characters. Uniqueness is enforced on frenzycreator.com — no two affiliates can create the same code.

Account Linking

When an affiliate registers on frenzycreator.com, they provide their exact CoinFrenzy username. This username is the bridge between both systems.

Registration Flow

  1. Affiliate enters their CoinFrenzy username on the frenzycreator.com signup form
  2. frenzycreator.com calls GET /affiliate/user/{username} to verify the username exists on CoinFrenzy — if non-200, signup is blocked
  3. frenzycreator.com stores this as casino_referral_code in the users table (status: pending)
  4. Admin approves the affiliate → frenzycreator.com calls POST /affiliate to register them on CoinFrenzy's side — if non-200, approval does not proceed
  5. After approval, the affiliate creates campaigns → each triggers POST /affiliate/promocode to sync the code to the main site
  6. No referral link is auto-generated at signup — all links and promo codes come from campaigns

NGR Data Feed (Weekly)

This is the core data integration. Every week, push Net Gaming Revenue data for every referred player who had activity.

What is NGR?

NGR = GGR − (|GGR| × 0.25)

GGR (Gross Gaming Revenue) = Purchases − Redemptions. The 25% deduction accounts for payment processing, fraud, provider revenue share, and licensing costs. Critically, the 25% is always subtracted from GGR based on the absolute value — this means:

  • Positive GGR (house won): 25% reduces the profit → NGR is lower → affiliate earns less commission
  • Negative GGR (player won): 25% increases the loss → NGR is more negative → larger negative carry-forward for the affiliate

This ensures CoinFrenzy is never absorbing losses that should reduce future affiliate payouts.

    "ngr_amount": 245.50,            // GGR − (|GGR| × 0.25) — can be negative
    "commission_percent": 10,          // Affiliate's rev share % (from users.rev_share_l1)
    "commission_amount": 24.55,       // NGR × (commission_percent / 100)
    "created_at": "2026-03-24T00:00:00Z"
  }
]

Calculation on CoinFrenzy.com side

// For each player in the reporting period:
GGR = Purchases − Redemptions
NGR = GGR − (|GGR| × 0.25)

// Example 1: House is profitable (player lost)
Purchases   = $12,500.00
Redemptions = $12,172.67
GGR         = $12,500 − $12,172.67 = +$327.33
NGR         = $327.33 − ($327.33 × 0.25) = $327.33 − $81.83 = +$245.50  ← push this

// Example 2: House lost (player won)
Purchases   = $500.00
Redemptions = $600.00
GGR         = $500 − $600 = −$100.00
NGR         = −$100 − ($100 × 0.25) = −$100 − $25 = −$125.00  ← push this
// Notice: The loss gets BIGGER (more negative) after the 25% deduction.
// This protects CoinFrenzy from absorbing costs on player wins.

// Simple rule:
// If GGR is positive  → NGR = GGR × 0.75  (house keeps less)
// If GGR is negative  → NGR = GGR × 1.25  (loss carried forward is larger)

Schedule: Every Monday at 00:00 UTC, covering the prior Monday–Sunday period.

Lightning Bolt Payout

This endpoint must be built by CoinFrenzy.com

When the admin clicks "Pay" on frenzycreator.com, we call a CoinFrenzy.com API endpoint that pushes SC (Sweepstakes Coin) to the affiliate's Lightning Bolt rewards balance — the lightning icon at the top of CoinFrenzy.com where players claim bonuses. SC is 1:1 USD equivalent. This is how affiliates get paid — they claim it from the Lightning Bolt directly inside their CoinFrenzy account.

Payout Flow

  1. Admin opens Payouts tab on frenzycreator.com/admin
  2. System shows each affiliate's unpaid balance (calculated from NGR data)
  3. Admin clicks "Send Payout" for one affiliate (or "Send All Payouts")
  4. frenzycreator.com calls POST /api/coinfrenzy/payout (proxied to …/frenzy-creator/payout) with the amount
  5. On success: payout is recorded in Supabase payouts table, balance resets
  6. On failure: admin sees error, nothing is recorded, balance unchanged

Bulk Payout

When admin clicks "Send All Payouts", frenzycreator.com will call the /affiliate/payout endpoint once per affiliate, sequentially. Each call is independent — if one fails, the others still succeed.

Deduplication

The reference_id field prevents double-payouts. If the same reference_id is sent twice, CoinFrenzy.com should reject the second call. Use our payout record ID as the reference.

Payout Calculation (Our Side)

This runs on frenzycreator.com — CoinFrenzy.com does not need to implement this. Documented for reference.

Cross-Player Netting

An affiliate's payout is based on the combined NGR of ALL their referred players. If Player A is +$500 and Player B is −$300, the affiliate's net NGR is $200. Commission is calculated on $200.

// Example: Affiliate coolcreator123 has 3 players this week
// NGR already calculated by CoinFrenzy.com: GGR − (|GGR| × 0.25)

Player A: GGR +$666.67  → NGR +$500.00   // $666.67 × 0.75
Player B: GGR -$240.00  → NGR -$300.00   // -$240 × 1.25 (loss amplified)
Player C: GGR +$266.67  → NGR +$200.00   // $266.67 × 0.75

Net NGR = $500 + (-$300) + $200 = $400
Commission (10% L1 rev share) = $400 × 0.10 = $40.00 SC to Lightning Bolt
// Note: 10% is the default L1 rate, stored per-affiliate in users.rev_share_l1
// If L2 is enabled, the parent affiliate also earns rev_share_l2 on sub-affiliate NGR

// If net NGR is negative:
Net NGR = -$200  →  Payout = $0, carry -$200 to next period
// Next week: net NGR $350, adjusted = $350 - $200 = $150
// Commission = $150 × 0.10 = $15.00

Level 2 Sub-Affiliates

Some affiliates can earn a secondary commission on their sub-affiliates' referred players. When L2 is enabled for a parent affiliate, they earn their rev_share_l2 rate on the NGR generated by all players referred by their sub-affiliates. This is managed entirely on our side — CoinFrenzy.com does not need to do anything different for L2 players.

Retry Strategy

Recommended Approach

For any 5xx error or network timeout, retry up to 3 times with exponential backoff: wait 1s, then 4s, then 16s. For 4xx errors, do NOT retry — fix the request first. For 409 on the Lightning Bolt credit call, treat as success (payout was already processed).

// Pseudocode: retry with exponential backoff
function postWithRetry(url, body, maxRetries = 3) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const res = await fetch(url, { method: 'POST', body, headers });
    
    if (res.status === 409) return { alreadyProcessed: true };
    if (res.ok) return await res.json();
    if (res.status >= 400 && res.status < 500) throw new Error('Client error');
    
    // 5xx → backoff and retry
    await sleep(1000 * Math.pow(4, attempt));
  }
  throw new Error('Max retries exceeded');
}

Idempotency

  • Lightning Bolt credit: The reference_id field ensures idempotency. If the same reference_id is sent twice, reject with 409.
  • Player registration: If a player with the same player_id already exists, Supabase returns 409. Treat as success.
  • NGR data: Each push should cover a unique report_period + player_id combination. If duplicated, Supabase will insert another row. Consider adding a unique constraint on (player_id, report_period) to prevent accidental duplicates.

Security

  • Supabase Service Role Key: provided separately — NEVER expose in client-side code. This key bypasses RLS.
  • CoinFrenzy API Key (Security Token): Used in Authorization: Bearer [API_KEY] header for all 8 CoinFrenzy API endpoints. Provided by CoinFrenzy dev team.
  • All requests must use HTTPS
  • All CoinFrenzy API endpoints must validate the API key
  • The /affiliate/payout endpoint must check reference_id for deduplication
  • The incoming webhook from CoinFrenzy should also be authenticated with the shared API key
  • Consider IP allowlisting for the Supabase service role key

Rate Limiting

Recommended Limits

Supabase: No rate limit concern for weekly batch pushes (typically <1000 records). For real-time player registration, expect single-digit requests per second — well within limits.

CoinFrenzy.com endpoints: Implement rate limiting on all 8 endpoints. Bulk payouts via /affiliate/payout are sent sequentially (one per affiliate).

CORS

All CoinFrenzy API endpoints will be called server-to-server or via authenticated client-side calls. CORS headers should allow requests from frenzycreator.com and www.frenzycreator.com origins.
